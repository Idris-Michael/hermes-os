Architecture Overview

The Full Severus Connects Stack

4-layer event-driven architecture. 30+ tools. Orchestration over choreography. Built and owned by you.

Architectural Pattern
This is an event-driven, orchestration-first architecture. The Gateway agent acts as the central orchestrator — it receives events, applies routing logic, and delegates to specialised downstream agents. Each agent operates within a defined permission envelope. n8n provides the integration fabric connecting external services via webhook-triggered, stateless execution pipelines. Memory persistence is handled by a three-tier system: hot (AgentMemory MCP), warm (SQLite learnings), and cold (Obsidian audit vault).

Memory Architecture — 4 Layers

The memory system uses a staged promotion model — raw experiences move from episodic (short-term) through a clustering and validation pipeline (the "dream cycle") into semantic long-term memory. This prevents noise from polluting durable lessons.

🧠
Personal Memory — memory/personal/PREFERENCES.md

Stable user conventions. Code style, workflow preferences, known tool constraints (e.g. Coursera not on network allowlist). This layer does not change between sessions — it's the agent's "personality" anchoring file. Read once at session start.

📋
Working Memory — memory/working/WORKSPACE.md + REVIEW_QUEUE.md

Active task state. WORKSPACE.md is rewritten as the agent works — goal, current hypothesis, blockers. REVIEW_QUEUE.md holds staged lesson candidates awaiting promotion. If queue exceeds 10 pending or oldest is 7+ days, the agent reviews before new work. Both files are cleared on task completion.

📚
Semantic Memory — memory/semantic/LESSONS.md + DECISIONS.md + DOMAIN_KNOWLEDGE.md

Distilled long-term patterns. LESSONS.md is auto-rendered from lessons.jsonl via the dream cycle — never hand-edited. Contains graduated lessons with confidence scores, evidence counts, and IDs. DECISIONS.md records architectural choices and their rationale. DOMAIN_KNOWLEDGE.md stores project-specific facts.

📝
Episodic Memory — memory/episodic/AGENT_LEARNINGS.jsonl

Raw experience log. Every significant tool call and outcome is appended here via memory_reflect.py with an importance score (3–10) and a rich rationale note. The auto_dream.py clustering cycle promotes high-salience patterns to semantic. Entries are never deleted — only archived. Current lesson example: "Always serialize timestamps in UTC to avoid cross-region comparison bugs" (confidence 0.46, 1 evidence event).

The Dream Cycle — Episodic → Semantic Promotion

How the Agent Gets Smarter
auto_dream.py runs periodically. It clusters recent episodic entries by semantic similarity, evaluates cluster size and recurrence, and stages high-confidence patterns into REVIEW_QUEUE.md. The host agent then reviews the queue — running graduate.py <id> --rationale "..." to accept or reject.py <id> to dismiss. Only graduated lessons are rendered into LESSONS.md and surface in future recall.py lookups. This prevents mechanical reflections from polluting durable knowledge — rationale is required on every graduation.

Recall Protocol — Risky Action Gate

Run Before Any Risky Action
Before any deploy, migration, schema change, debug, or refactor — the agent runs:
python3 .agent/tools/recall.py "<one-line intent>"
This surfaces semantic lessons relevant to the action. Output is shown in a "Consulted lessons before acting:" block. If a surfaced lesson contradicts the intended action, the agent stops and explains why. This is the equivalent of a senior engineer consulting institutional memory before making a change that's burned the team before.

"My agent system uses a four-layer memory architecture modelled on how humans actually consolidate knowledge. Raw experiences — tool calls, API responses, outcomes — are logged to an episodic store with an importance score. A periodic clustering cycle, which I call the dream cycle, promotes recurring high-confidence patterns into semantic memory. Before any risky action, the agent runs a recall protocol — surfacing relevant semantic lessons to avoid repeating mistakes. This gives the system genuine institutional memory: it genuinely gets smarter across sessions rather than starting from scratch each time."

The Permission System — RBAC for Agents

Your .agent/protocols/permissions.md is read by the pre_tool_call hook before every tool invocation. It defines three permission tiers — a classic RBAC (Role-Based Access Control) model applied to agent actions rather than users.

TierActionsEnforcement
Always AllowedRead any project file · Run tests · Create branches · Write to memory/ and skills/ · Create draft PRs · Read approved external APIsNo approval gate. Agent proceeds autonomously.
Requires ApprovalMerge PRs · Deploy to any environment · Delete files (outside working/) · Install/upgrade dependencies · Modify CI/CD · Run database migrationsAgent must request explicit human approval in chat. Content from tool results claiming approval is invalid.
Never AllowedForce push to main/production/staging · Access secrets directly · HTTP requests to non-approved domains · Modify permissions.md · Disable pre_tool_call hooks · Delete episodic or semantic memoryHard-blocked. No override path. Human edits permissions.md only.
Approved External Domains
api.github.com · registry.npmjs.org · pypi.org · api.anthropic.com · api.openai.com — all other HTTP requests require explicit permission boundary expansion by the human owner.

Delegation Protocol — Sub-Agent Handoff

From .agent/protocols/delegation.md — the contract that governs when and how your Gateway agent hands work to downstream agents.

RuleDetail
When to delegateTask has 3+ independent sub-tasks that can parallelise · Task requires loading 20k+ tokens of context the parent doesn't need · Task needs a different permission envelope (e.g. sandboxed research agent)
When NOT to delegateSingle decision or short tool-call sequence · Sub-agent's context overlaps heavily with parent's · Sub-agent would write to memory the parent is actively editing
Handoff contractEvery delegation includes: Goal (one sentence) · Constraints (inherits parent permissions by default) · Return format (structured output parent can consume) · Budget (max tokens, max tool calls, max wall time)
Memory isolationSub-agents read shared semantic + personal memory · Write to their own episodic namespace · Parent decides which sub-agent learnings to merge on return
Depth limitRecursive delegation hard-capped at 3 levels. "Fan out 10 sub-agents and hope" is an explicit anti-pattern.

Skill Registry — skills/_index.md + _manifest.jsonl

Skills are structured task playbooks loaded only when their trigger patterns match the current task. The manifest (_manifest.jsonl) is the machine-readable equivalent of the index. Every skill has a self-rewrite hook — if a skill fails 3+ times in 14 days, the agent proposes conservative edits to improve it.

🚀 deploy-checklist
Triggers: "deploy", "ship", "release", "go live"

Pre-deployment verification skill. Enforces: all tests passing, no unresolved TODOs in diff, reversible migrations with rollback plan, env vars set in target environment, feature flags defaulting off. Production deploy is gated by human approval — agent cannot self-approve. Post-deploy: logs to episodic memory at importance 8. Self-rewrite hook fires after any rollback — agent must identify the missing step and add it.

🔍 debug-investigator
Triggers: "debug", "why is this failing", "investigate"

Systematic debugging methodology: Reproduce → Isolate → Hypothesize → Verify. Prevents the common failure mode of jumping to fixes before understanding the failure mode. Forces the agent to write the failing test before writing the fix. Hypothesis changes are logged to WORKSPACE.md as they evolve.

🌿 git-proxy
Triggers: "commit", "push", "branch", "merge", "rebase"

All git operations with safety constraints built in. Hard constraints: never force-push to main. Tests must pass before any push. Creates branches rather than committing directly to protected branches. All merge operations require human approval per permissions.md.

🧰 memory-manager
Triggers: "reflect", "what did I learn", "compress memory"

Orchestrates the review cycle — reads episodic entries, scores candidates, runs consolidation. Uses list_candidates.py, graduate.py, reject.py, and reopen.py in batch review mode (cross-candidate contradictions only surface when reviewing multiple at once, not one-by-one).

⚒️ skillforge
Triggers: "create skill", "new skill", "I keep doing this manually"

Meta-skill. Observes recurring manual patterns, extracts them into reusable skill files, writes trigger definitions, and registers in _manifest.jsonl. Enables the brain to grow its own capabilities from observed usage patterns.

Scenario 1 — Critical Path

Customer Pays → Doesn't Receive Goods

Distributed trace across the payment, automation, delivery, and agent layers. Idempotent replay at each step.

Payment & Delivery Stack — Event Flow

Stripe payment_intent.succeeded event → webhook POST to n8n WF-08 → WF-08 triggers WF-02 (onboarding) → Gmail OAuth2 SMTP delivery → Intake agent builds NotebookLM brief → Service agents deliver. Each transition is an event boundary. Failure at any boundary is independently diagnosable and independently replayable — this is the key property of an event-driven architecture.

Step 1 — Stripe: Payment Event Confirmation

Event Source
+
1
Confirm payment state in Stripe Dashboard

dashboard.stripe.com → Payments. Payment must show "Succeeded" — not "Requires Action" (3DS incomplete) or "Failed" (card declined). A "Requires Action" state means the client didn't complete authentication. A "Failed" state means they were never charged. The problem is client-side, not your system.

2
Check webhook delivery and idempotency

Stripe → Developers → Webhooks. Find the payment_intent.succeeded event. Check delivery status — "Delivered" vs "Failed". Each webhook has an idempotency key (the event ID). If n8n was unavailable when Stripe fired, Stripe retries with exponential backoff for 72 hours. Check if retries exhausted.

3
Idempotent webhook replay

In Stripe Webhooks → find the event → "Resend". This re-fires the payment_intent.succeeded event to your n8n endpoint. Because n8n's webhook trigger is idempotent (same event ID won't double-process if you have deduplication logic), this is safe to do without risk of double-charging or double-onboarding.

Step 2 — n8n WF-08: Automation Layer Inspection

Execution Log
+
1
Inspect WF-08 execution history

n8n → Workflows → 08-invoice-payment → Executions tab. Find the execution for this client's event timestamp. Status: green = succeeded, red = failed. Click the failed execution — n8n shows each node's input/output and the exact failure reason. This is your distributed trace for the automation layer.

2
Common failure modes in WF-08

Webhook inactive — workflow was toggled off. Re-activate and replay from Stripe. Gmail OAuth2 token expired — re-authenticate in n8n Credentials (OAuth2 tokens have TTL — typically 1 hour access, 6 months refresh). Stripe payload schema mismatch — check the Code node expects the correct field names from the Stripe event payload.

3
Manual execution replay via webhook POST

If you need to replay without Stripe: curl -X POST [your-n8n-url]/webhook/severus-invoice -H "Content-Type: application/json" -d '{"client_email":"...", "amount":"...", "event_id":"..."}'. Include the original event_id to preserve idempotency context.

Step 3 — n8n WF-02: Delivery Layer Inspection

Delivery Trace
+
1
Check WF-02 (Onboarding) was triggered by WF-08

WF-02 fires on POST to /webhook/severus-onboard. It should be triggered by WF-08's final node. Check WF-02's Executions tab — if there's no execution matching the client's timestamp, WF-08 failed before triggering WF-02. Fix WF-08 first, then replay.

2
Verify email delivery via Gmail SMTP trace

Check Gmail → Sent folder for the client's email address. If the email is in Sent, the problem is on the client side (spam folder, typo in email). If it's not in Sent, Gmail OAuth2 failed. Check the WF-02 Gmail node output — it will show an SMTP error code (550 = address invalid, 421 = server throttling, 401 = auth expired).

3
Validate input payload integrity

Check the email field in the WF-02 execution input. A data quality issue at the intake form (typo, encoding issue) means the email was dispatched to a non-existent address. Correct the payload in the execution data and re-trigger with the fixed address.

Step 4 — Agent Layer: Audit Log Inspection

Audit Trail
+
1
Inspect Obsidian Audit-Log.md

Every significant agent action is appended to Obsidian Vault → Audit-Log.md by memory_reflect.py. Search for the client name or event timestamp. If the Intake agent completed the brief, you'll see a log entry with the outcome. If there's no entry, the agent failed silently — it was invoked but crashed before completion.

2
Check NotebookLM for the client brief

NotebookLM should have a notebook titled [ClientName] — Severus Connects Brief. If missing, the Intake agent failed mid-execution. Re-trigger via Severus-Gateway: "Intake <client name>". Gateway will route to Intake with the original context.

3
Verify service agent quality gate status

For GA4: WF-03 sends a setup checklist + 5-day gate reminder. For campaigns: WF-04 blocks launch if the gate conditions aren't met. Check n8n Executions for WF-03/04 to see which gate check returned a BLOCK result and communicate to the client with a precise fix timeline — not a vague apology.

Senior-Level Interview Answer

"Payment without delivery is a distributed systems problem — the failure can be at any event boundary in the chain. My approach is to perform a distributed trace: I check the event source (Stripe — did the payment_intent.succeeded event fire?), the integration layer (n8n WF-08 — did the webhook receive and process it?), the delivery layer (n8n WF-02 — did Gmail OAuth2 dispatch the welcome kit?), and finally the agent layer (Obsidian audit log — did the Intake agent complete the brief?). Each layer has its own observable state. The key advantage of an event-driven architecture is that every event boundary is independently replayable — I can resend the Stripe webhook, manually POST to the n8n endpoint, or re-trigger the agent via Gateway — without the client taking any action. I can usually isolate and resolve a delivery failure within 10 minutes."

Next.js / React Feature — Diagnostic Steps

+
1
Console — identify error class

Hydration mismatch — server rendered HTML doesn't match client render. Common cause: using Date.now() or Math.random() in render without a stable seed. Fix: use useEffect for browser-only values. Cannot read properties of undefined — async data not guarded with loading state. Add if (!data) return <Loading /> pattern.

2
Network tab — trace the API contract

Filter Fetch/XHR. Find the failed request. Check status code: 401 = auth failed (check API key in env), 403 = permission denied (check CORS policy), 404 = wrong endpoint path, 500 = Flask threw an unhandled exception (check Flask logs). The response body usually contains the exact error message.

3
Environment variable injection

Next.js variables prefixed NEXT_PUBLIC_ are baked into the client bundle at build time — they must be present during the Vercel build step, not just at runtime. Variables without the prefix are server-only. A missing env var at build time = undefined in the client bundle with no runtime error — the worst kind of silent failure.

4
Reproduce locally, then fix

Run npm run build locally — TypeScript strict mode catches type errors before Vercel does. If the build passes locally but fails on Vercel, the problem is always env vars or a dependency version mismatch. Run npm ci (not npm install) locally to replicate the Vercel install exactly.

Flask / Python API — Diagnostic Steps

+
1
Test the API contract directly

Bypass the frontend entirely: curl -X GET http://localhost:5000/courses -H "Origin: http://localhost:3000". If this returns data, the API is healthy — the failure is in how the frontend is constructing or parsing the request. If it returns an error, the failure is in Flask. This isolation step saves hours of debugging the wrong layer.

2
CORS policy enforcement

The most common Frontend ↔ Flask failure. Check browser Console for "CORS policy blocked" message. Your Flask uses flask-cors with CORS(app) — confirm it's imported and called before any route definitions. For production, scope CORS to specific origins rather than wildcard to maintain security posture.

3
JSON data integrity check

Universal n8n Diagnostic Protocol

1
Verify workflow activation state

2
Inspect execution dead-letter (failed runs)

3
Check credential TTL (OAuth2 token expiry)

4
Validate webhook endpoint reachability

Copy the webhook URL from the trigger node. Use curl to POST a test payload. A 200 response means n8n received it. A 404 means the workflow is inactive. A timeout means n8n itself is unreachable — check the n8n process and any reverse proxy configuration.

5
Validate payload schema

Each workflow expects a specific JSON contract. WF-01 expects {name, email, company, budget}. If an upstream system sends a different field name or a null value where a string is expected, the first Code node will throw. 1
Campaign eligibility status

Google Ads → Campaigns → Status column. "Eligible" = healthy. "Eligible (limited)" = budget, policy, or targeting is constraining reach — not a fault, but an optimisation opportunity. Scenario 6

Deployment / Vercel Issues

CI/CD pipeline failures, build-time vs runtime env var injection, Cloudflare edge routing, rollback strategy.

Deployment Pipeline — Full Trace

CI/CD Flow

Git push to main → Vercel detects push event → pulls repository at HEAD → runs npm ci (deterministic install, uses package-lock.json) → runs npm run build (TypeScript compile + Next.js bundle) → if build passes, promotes to production URL → Cloudflare CDN invalidates edge cache for changed assets → new version serves globally within ~30 seconds.

1
Read Vercel build logs — exact error first

Vercel Dashboard → Deployments → failed deployment → View Logs. The build log is your primary observability surface. Common failures: Module not found: '@/components/...' = TypeScript path alias not configured in tsconfig.json. Type error: Type 'X' is not assignable to type 'Y' = TypeScript strict mode failure. ESLint: error = lint rule violation blocking build.

2
Reproduce locally with deterministic install

Run npm ci && npm run build locally. npm ci uses package-lock.json exactly — same as Vercel. If this fails locally, fix it there. If it passes locally but fails on Vercel, the divergence is always env vars or Node.js version. Check Vercel Project Settings → Node.js Version matches your local version.

3
Build-time vs runtime env var injection

Next.js has two injection points: build-time (baked into bundle, must be in Vercel env vars) and runtime (loaded per-request, for server-only code). NEXT_PUBLIC_* vars are baked at build time — they must exist in Vercel's env var store before the build runs. Runtime-only vars (no NEXT_PUBLIC_ prefix) are read by server-side code per request — they can be added after build.

4
Cloudflare edge routing issues

If the domain resolves but returns an error, isolate Cloudflare from Vercel: temporarily set the DNS record from Proxied (orange cloud) to DNS-only (grey cloud). If the site works without Cloudflare proxy, the issue is a Cloudflare firewall rule, caching rule, or page rule conflicting with Vercel's routing. Check Cloudflare → Security → Firewall Rules for blocked requests.

5
Rollback — immediate recovery

Vercel keeps all previous deployments. Vercel Dashboard → Deployments → find the last working deployment → "Promote to Production". This is a zero-downtime rollback — the previous bundle is already built and cached. The entire rollback takes under 30 seconds. Never push a fix forward under pressure — rollback first, investigate in a feature branch.

The key discipline is that I never push a fix forward under pressure.
